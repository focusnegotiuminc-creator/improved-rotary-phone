## Screenshot-derived protocol requirements

### Architecture of loop

The uploaded navigation protocol defines the loop as:

1. Local Update — introduce specific constraint or query.
2. Global Consistency Check — verify alignment with the operator's topological structure.
3. Local Refinement — adjust query, spatial coordinate, or container assignment.
4. Global Collapse — stop querying and integrate the result once stable resolution is achieved.

### Phase and temporal position

- 0 degrees = initiation / open operational field.
- 90 degrees = compression / multiple constraints active.
- 180 degrees = collapse / decision point.
- 270 degrees = expansion / integration before next cycle.

### Mode selection

- Neutral: initial assessment.
- Local: specialized execution.
- Non-local: pattern recognition and meta-analysis.
- Recursive: loop monitoring and quality control.
- Navigational: transition between AI entities or phase changes.
- Void: rest, incubation, and reset.

### Container map

The protocol treats model/tool containers as coordinate resources rather than authorities. The software must represent containers as configurable capabilities, not fixed truths.

### Failure modes

| Failure | Software handling |
|---|---|
| Drift | reset to neutral; perform coordinate check |
| Compression Lock | reduce temporal pressure; allow void mode |
| Attractor Capture | hard container shift; alter eta/theta |
| Identity Hijacking | agency check: AI is mirror, not oracle |
| Premature Collapse | ask if resolution is self-evident or just available |

### Advanced operations

- Nested Loop: use one AI/container to design the query for another.
- Parallel Fold: run multiple positions simultaneously, then collapse at global check.
- Void Insert: intentional break with no query for a time period.
- Sovereignty Check: external systems are coordinates, not destinations.

## Implementation requirements

- Use deterministic enums for phase and mode.
- Log every loop stage.
- Detect and report failure modes.
- Keep human approval as the final gate.
- Keep recoding changes branch-based.
- Keep trading or monetization systems separate from the Operator OS kernel.
